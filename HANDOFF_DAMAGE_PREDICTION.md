# 핸드오프 — 허수아비 기준 대미지 예측 시스템

> 작성: 2026-04-30 (오전 세션 / 오후 인계용)
> 대상: 오후에 이어 작업할 Claude
> 상태: **모델 합의 완료 / 사용자가 데이터 수집 중 / 데이터 도착 즉시 구현 시작 가능**
>
> 이전 핸드오프 `HANDOFF_CONDITIONAL_DAMAGE.md`는 폐기됨 — 본 문서가 그 상위 집합이다.

---

## 1. 목표

사용자가 입력한 **스탯 + 스킬계수**만으로 "이 스킬 1타가 허수아비에 들어갈 **평균 대미지**"를 예측한다.

부수 효과: 백어택·근거리·상태대미지 같은 조건부 옵션은 분모를 가정하지 않고 **사용자가 옵션 ON/OFF 두 측정으로 곱셈 배율을 직접 도출**하면 끝 (별도 모델링 불필요).

---

## 2. 핵심 모델

라테일 대미지 공식이 BP 공식과 같은 곱셈 구조를 공유한다는 가정 하에, **고정 타겟(허수아비)** 기준:

```
허수아비 평균 단발 대미지 = C × BP × (스킬계수 / 100)
```

- `BP` = `calculateBattlePower(stats)` 결과 (변경 없음, 회귀 학습값 그대로)
- `스킬계수` = 게임 내 스킬 정보 % 값 (예: 4500% → 45)
- `C` = 허수아비 타겟계수 — 사용자 측정 데이터로 회귀 추정

**검증 가설**: 동일 직업군(P/M)에서 C는 거의 일정. 만약 사용자 간 편차가 ±5% 이내면 글로벌 상수 두 개(`C_PHYSICAL`, `C_MAGIC`)로 충분. 더 크면 캐릭별 1회 캘리브레이션 필요.

---

## 3. 기준 타겟 = 허수아비 (수련의방 max 아님)

| | 수련의방 (max 대미지) | 허수아비 (누적 대미지) |
|---|---|---|
| 측정값 | max 단발 (최대 롤 + 크리티컬 정점) | 평균 누적 (RNG 자연 흡수) |
| 일관성 | 운에 의존, 본인 최고치 | 다회 누적 → 평균 수렴 |
| 백어택 등 조건부 | 조건 충족된 단발만 | 적용률까지 자연 반영 |
| BP 공식 적합성 | max는 별도 공식 필요 | **BP가 표현하는 기댓값과 직접 비례** |

→ **본 시스템의 모든 대미지 입력/예측은 허수아비 기준으로 통일.** 수련의방 값은 입력 받지 않는다 (혼동 방지).

## 4. 사용자가 제공할 데이터 포맷

집에 와서 사용자가 직접 측정하고 가져올 데이터. 아래 표 형식이면 바로 회귀 가능.

### 4.1 캐릭터 단위 (1행 1캐릭터)

| 필드 | 예시 | 비고 |
|---|---|---|
| 캐릭명/직업 | `다크체이서` | 식별용 |
| type | `P` 또는 `M` | 물리/마법 |
| 주스탯, 공격력, 관통, 크댐, 최소뎀, 최대뎀, 고댐 | T창 표시값 | 현재 `StatInputForm` 필드 그대로 |
| 일몬추, 보몬추, 일몬지, 보몬지, 근마효율 | T창 표시값 | 〃 |
| 기본_* (8개) | 추가 세부정보 +값 | 장비 % 환산용 (현재 폼과 동일) |

### 4.2 측정 단위 (1행 1측정 — 캐릭터당 **2~3개 이상** 권장)

| 필드 | 예시 | 비고 |
|---|---|---|
| 캐릭명 | `다크체이서` | 위 표 참조 |
| 스킬명 | `메테오` | 식별용 |
| 스킬계수(%) | `4500` | 게임 스킬창의 % 값 |
| 시전 횟수 | `30` | 누적 측정에 쓴 횟수 |
| 누적 대미지 | `370,200,000` | 허수아비에 친 총합 |
| 평균 단발 대미지 | `12,340,000` | = 누적 ÷ 횟수 (자동 계산해도 됨) |
| 크리티컬 가능 여부 | `Y` | N이면 비크리만 |
| 백어택 자세 | `Y/N/혼합` | 옵션 효과 측정 시 |
| 근거리 여부 | `Y/N` | 〃 |
| 상태이상 부여 | `Y/N` | 〃 |
| 측정 일자 | `2026-04-30` | 패치/버프 변동 추적용 |

### 4.3 검증을 위한 최소 권장 분량

- **캐릭터 1명당 최소 2개 이상의 다른 스킬계수** 측정 → 모델이 진짜 선형(`damage = C × BP × coef`)인지 검증.
- **물리/마법 각 1명 이상** → C가 type별로 다른지 확인.
- **여유되면 2~3명** → C가 캐릭간 일관된지 확인 (글로벌 상수 vs 캐릭별 캘리브레이션 결정).

데이터 1건만 와도 캘리브레이션은 가능. 단 모델 검증·일반화 가능성은 떨어진다.

## 5. C 추정 절차 (오후 Claude가 할 일)

데이터 도착 시 다음 순서로 진행:

### Step 1. 데이터 정합성 확인

- 평균 단발 대미지 = 누적 ÷ 횟수 검산
- 시전 횟수가 너무 적으면 (≤10) 분산 큼 → 사용자에게 추가 측정 요청 고려
- 비크리만 측정인지, 크리 포함 평균인지 명확히 파악
  - **비크리 only**: BP 공식의 `critMultiplier`를 빼고 비교해야 함 → 별도 baseline 계산 필요
  - **크리 평균 포함**: BP 공식 그대로 사용 가능 (권장)

### Step 2. C 회귀

각 측정 점마다:

```
C_i = 평균 단발 대미지_i / (BP_i × 스킬계수_i / 100)
```

- 모든 `C_i`가 ±5% 이내로 모이면 → 모델 OK, `C_global = mean(C_i)` 사용
- 큰 편차 있으면 원인 추적:
  - 스킬마다 추가 보정이 있는지 (스킬 자체 조정 계수)
  - 백어택 등 조건부 옵션이 일부 측정에만 적용됐는지
  - 수치 자릿수/환산 오류 (사용자에게 재확인)

### Step 3. type별 분리 검증

- 물리 측정과 마법 측정의 `C` 비교 → `MAGIC_PARAMS`의 K1 보정처럼 별개 상수일 수 있음
- 필요 시 `C_PHYSICAL`, `C_MAGIC` 두 상수로 분리

### Step 4. 코드에 상수 박제 또는 캘리브레이션 모드 결정

| 시나리오 | 처리 방법 |
|---|---|
| C가 ±5% 이내로 일정 | `battlePower.js`에 `C_PHYSICAL`, `C_MAGIC` 상수로 박제 |
| C가 캐릭간 ±10%+ 편차 | 캐릭터별 캘리브레이션 입력 UI 추가 (사용자가 1회 측정값 입력 → C 자동 도출 → localStorage 저장) |
| 스킬별로도 C 차이 | 스킬 카테고리(타격/지속/광역) 별 보정 계수 도입 검토 |

## 6. UI 설계

### 6.1 스킬계수 입력

`StatInputForm.vue` 또는 별도 컴포넌트(`SkillDamagePredictor.vue` 신설 권장)에 단일 input:

- 라벨: "스킬계수 (%)"
- placeholder: "예: 4500"
- 단일 값 — 여러 스킬을 비교하려면 추가 행 입력 가능하게 (선택)

### 6.2 예측 대미지 표시

`ResultDisplay.vue`에 새 카드:

- **예상 평균 단발 대미지** = `C × BP × 계수/100`
- 보조 정보: 최소/최대 추정 범위 (현재 스탯의 최소뎀~최대뎀 표시값 비율로 계산)
- 백어택/근거리/상태 옵션 ON/OFF 토글이 있으면 각 옵션별 곱셈 배율 적용한 값도 함께

### 6.3 캘리브레이션 모드 (Step 4에서 캐릭별 캘리브레이션이 필요하다고 판단 시)

- 별도 섹션: "내 캐릭터 캘리브레이션"
- 입력: 측정 스킬계수 + 측정 평균 단발 대미지
- 자동 도출: `C_user = 측정값 / (BP × 계수/100)`
- 저장: `useCharacterStorage`로 캐릭터 단위 저장
- 표시: "이 캐릭의 C = 0.0XXX (글로벌 평균 대비 ±X.X%)"

### 6.4 백어택 등 조건부 옵션 통합 (원래 문제 해결)

이전 핸드오프의 별도 분모 추측은 **불필요해짐**. 대신:

- 사용자가 같은 캐릭/스킬을 옵션 ON/OFF 두 번 측정
- 비율 = `대미지_ON / 대미지_OFF` → 그게 곧 그 옵션의 진짜 곱셈 배율
- UI에서 옵션 ON 토글 시 이 배율을 예측 대미지에 곱해서 표시
- 사용자가 측정값 입력하지 않으면 옵션 토글은 비활성화 (값 모르므로 추측 안 함)

## 7. 구현 가이드 (코드 위치)

### 7.1 `src/utils/battlePower.js` (수정 최소화)

- 기존 `calculateBattlePower(stats)` **그대로 유지** (회귀 학습값 보존, 호환성 유지)
- 새 export 추가:
  ```js
  export const C_PHYSICAL = ???;  // 데이터 도착 후 결정
  export const C_MAGIC = ???;

  export function predictDummyDamage(stats, skillCoefPercent, options = {}) {
    const bp = calculateBattlePower(stats);
    const C = stats.type === 'M' ? C_MAGIC : C_PHYSICAL;
    const userC = options.userC ?? C;
    return userC * bp * (skillCoefPercent / 100);
  }
  ```
- 옵션 곱셈 배율은 별도 인자로:
  ```js
  predictDummyDamage(stats, coef, {
    userC: 0.0123,
    multipliers: { 백어택: 1.30, 근거리: 1.15 },
  });
  ```

### 7.2 새 컴포넌트 `src/components/SkillDamagePredictor.vue` (권장)

- 메인 레이아웃에 새 섹션으로 추가
- 스킬계수 입력 + 예측 대미지 표시 + 옵션 토글 + 캘리브레이션 입력 (필요 시)
- props: 현재 캐릭터 스탯
- `App.vue`에서 `<SkillDamagePredictor :stats="currentStats" />` 형태로 마운트

### 7.3 `src/composables/useCharacterStorage.js`

- 캐릭터 객체에 새 필드 추가:
  - `userC?: number` — 캐릭별 캘리브레이션 결과 (있으면 글로벌 C 대신 사용)
  - `conditionalMultipliers?: { 백어택: number, 근거리: number, 상태대미지: number }` — 옵션 ON/OFF 측정으로 도출한 곱셈 배율
- 누락 필드는 폴백으로 처리 (기존 저장 호환)

### 7.4 `tests/`

- 기존 BP 회귀 테스트 절대 깨지지 않는지 확인 (회귀 방지)
- 새 함수 단위 테스트:
  - `predictDummyDamage` — BP × coef × C 계산 정확성
  - 옵션 곱셈 배율 적용
  - userC 우선순위

## 8. 데이터 도착 후 단계별 작업 순서

1. **데이터 검증** — 4.2 표 형식 맞는지, 시전 횟수 충분한지, 측정 조건 명확한지 사용자에게 빠르게 확인

6. **빌드/실행** — `npm run dev` 부탁드리고 사용자 환경에서 실측값 vs 예측값 비교
7. **커밋 단위** — "predictDummyDamage 추가" → "UI 컴포넌트" → "캘리브레이션 모드" → "조건부 곱셈 배율" 순으로 쪼개서

## 9. 모델이 맞는지 검증 체크리스트

데이터 도착 후 다음을 확인:

- [ ] 같은 캐릭에서 다른 스킬계수의 C_i 편차 < 5% (선형성 검증)
- [ ] 다른 캐릭(같은 type) 간 C 편차 < 5% (일반화 검증)
- [ ] 물리 vs 마법 type별 C 차이가 PHYSICAL_PARAMS/MAGIC_PARAMS 비율과 정합 (모델 자기일관성)
- [ ] 백어택 ON/OFF 측정값 비율이 게임 내 표시값(예: 백어택 30%)과 정합 (예: ON/OFF ≈ 1.30)

전부 통과하면 모델 박제. 하나라도 어긋나면 5번 Step 2 원인 추적.

## 10. 주의 사항

- **`PHYSICAL_PARAMS`, `MAGIC_PARAMS` 절대 수정 금지** — 56건 회귀 학습 결과. 본 작업과 무관.
- **`calculateBattlePower(stats)` 시그니처 변경 금지** — 기존 호출처 다수, 호환 깨짐.
- **수련의방 max 입력 받지 말 것** — 5번에서 평균 대미지로 정의했고, max는 별도 공식.
- **사용자는 한국어 UI 선호.** 영문 라벨/메시지 추가 금지.
- **단발 측정 데이터로 C 결정 금지** — 분산 너무 큼. 누적/평균만 사용.
- **크리티컬 처리** — BP 공식이 이미 크리티컬 기댓값을 포함한 곱셈 구조라, 측정도 "크리 포함 평균"으로 통일하는 게 정합. 비크리 only 측정은 별도 처리 필요 → 사용자에게 측정 모드 명확화 요청.

## 11. 참고 — 현재 BP 공식 (변경 없음)

```
attackBase = K0×주스탯 + K1×공격력 + K2×고댐 + K_mon×(일몬추+보몬추)

BP = base ×
     attackBase ×
     (1 + 크댐/D_crit) ×
     (1 + (최소뎀+최대뎀)/D_dmg) ×
     (1 + (일몬지+보몬지)/D_dom) ×
     (1 + 근마효율 × K_geunma) ×
     (1 + 관통/D_pen)
```

본 작업은 이 BP를 입력으로 받아 `damage = C × BP × coef/100`만 추가로 계산. BP 공식 자체는 건드리지 않음.
